# Password Reset

Tokens are generated on the **Django backend**. The reset email is sent by **EmailJS on the frontend**.

---

## What the backend does

1. User submits their email → backend looks up the account, generates a `uid` + `token`, returns them in the response.
2. Frontend constructs the reset link and fires EmailJS: `/auth/reset-password?uid=<uid>&token=<token>`.
3. User lands on that page, submits a new password → backend validates the token, sets the new password, invalidates all existing auth tokens (forces re-login).

The token is generated by Django's `default_token_generator`. It is tied to the user's current password hash and last login time, so it **auto-invalidates** the moment the password is changed — no manual expiry needed.

---

## API Endpoints

### 1. Request a reset link

```
POST /api/auth/request-password-reset/
```

**Auth:** None (public)

**Request body:**
```json
{ "email": "user@example.com" }
```

**Response when account exists — `200`:**
```json
{
  "detail": "If an account with that email exists, a reset link has been sent.",
  "uid": "MQ",
  "token": "abc123-xyz"
}
```

**Response when account does not exist — `200`:**
```json
{ "detail": "If an account with that email exists, a reset link has been sent." }
```

> The message is the same in both cases. The absence of `uid`/`token` is the only difference — the frontend checks for their presence before calling EmailJS. The user always sees the neutral message.

---

### 2. Confirm the reset (set new password)

```
POST /api/auth/confirm-password-reset/
```

**Auth:** None (public)

**Request body:**
```json
{
  "uid":          "MQ",
  "token":        "abc123-xyz",
  "new_password": "NewSecurePassword1!"
}
```

`uid` and `token` come directly from the URL query params the user clicked in the email.

**Success response — `200`:**
```json
{ "detail": "Password reset successfully. Please log in with your new password." }
```

**Error response — `400`** (invalid/expired token, password too weak, missing fields):
```json
{ "detail": "Invalid or expired reset link." }
```

> On success, all existing auth tokens for that user are deleted. The user must log in again with the new password.

## Full flow end-to-end

```
User on /auth/login clicks "Forgot password?"
  → navigates to /auth/forgot-password

User enters email and submits
  → POST /api/auth/request-password-reset/ { email }
  ← 200 { detail, uid?, token? }
  → If uid + token present: emailjs.send('password_reset_email', { to_email, user_name, reset_link })
  → Show user: "If an account with that email exists, a reset link has been sent."

User clicks link in email
  → lands on /auth/reset-password?uid=MQ&token=abc123-xyz
  → page reads uid and token from URL

User enters new password and submits
  → POST /api/auth/confirm-password-reset/ { uid, token, new_password }
  ← 200 → redirect to /auth/login with success message
  ← 400 → show error (link expired, already used, password too weak)
```

## EmailJS Setup

**Template: `password_reset_email`**
- To Email: `{{to_email}}`
- Variables used in body: `{{user_name}}`, `{{reset_link}}`

**Required frontend env vars:**
```
REACT_APP_EMAILJS_PUBLIC_KEY=
REACT_APP_EMAILJS_SERVICE_ID=
REACT_APP_EMAILJS_TEMPLATE_RESET=
```

## What the frontend needs to build

### Pages

#### `/auth/forgot-password`
- Single email input field
- On submit: call `requestPasswordReset(email)` from `AuthContext`
- Always show the neutral message after submit regardless of what came back
- EmailJS is called internally by `AuthContext` — the page doesn't need to know about it

#### `/auth/reset-password`
- Reads `uid` and `token` from URL search params on mount
- If either is missing, show an error and link back to `/auth/login`
- New password input + confirm password input
- On submit: call `authService.confirmPasswordReset(uid, token, newPassword)` (or add to AuthContext)
- On success: show confirmation message and redirect to `/auth/login`
- On error: show the `detail` message from the response

### Service methods (`frontend/src/services/auth.js`)

`requestPasswordReset` is already implemented. Add `confirmPasswordReset`:

```javascript
confirmPasswordReset: async (uid, token, newPassword) => {
  const response = await authApi.post('/auth/confirm-password-reset/', {
    uid,
    token,
    new_password: newPassword,
  });
  return response.data;
},
```

Note the field name: `new_password` (snake_case) — that is what the backend expects.

### "Forgot password?" entry point

Add a **"Forgot password?"** link on the login page below the password field.
Link destination: `/auth/forgot-password`

## Notes

- **Token lifetime:** Django's default is 3 days (`PASSWORD_RESET_TIMEOUT` setting). Adjust in `backend/lamla/settings.py` if you want stricter expiry (e.g. `PASSWORD_RESET_TIMEOUT = 86400` for 24 hours).
- **One-time use:** The token is automatically invalidated after a successful reset because the password hash changes.
- **Rate limiting:** The request endpoint is throttled at 5 requests/hour per IP.
- **Email enumeration:** The response shape differs when the account exists (`uid`+`token` present) vs not (only `detail`). This is an accepted tradeoff — the rate limiter makes brute-force enumeration impractical.
