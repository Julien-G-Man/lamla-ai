# Password Reset

The backend is fully implemented. This doc tells the frontend dev exactly what to build.

---

## What the backend does

1. User submits their email → backend looks up the account, sends a reset link via email (GAS → Brevo → Resend → SMTP fallback chain).
2. The email contains a link to `/auth/reset-password?uid=<uid>&token=<token>`.
3. User lands on that page, submits a new password → backend validates the token, sets the new password, and invalidates all existing auth tokens (forces re-login).

The token is generated by Django's `default_token_generator`. It is tied to the user's current password hash and last login time, so it **auto-invalidates** the moment the password is changed — no manual expiry needed.

---

## API Endpoints

### 1. Request a reset link

```
POST /api/auth/request-password-reset/
```

**Auth:** None (public)

**Request body:**
```json
{ "email": "user@example.com" }
```

**Response — always `200` regardless of whether the email exists:**
```json
{ "detail": "If an account with that email exists, a reset link has been sent." }
```

> This is intentional. Returning an error when the email is unknown would let attackers enumerate registered accounts. Always show the same message to the user.

---

### 2. Confirm the reset (set new password)

```
POST /api/auth/confirm-password-reset/
```

**Auth:** None (public)

**Request body:**
```json
{
  "uid":          "MQ",
  "token":        "abc123-xyz",
  "new_password": "NewSecurePassword1!"
}
```

`uid` and `token` come directly from the URL query params the user clicked in the email.

**Success response — `200`:**
```json
{ "detail": "Password reset successfully. Please log in with your new password." }
```

**Error response — `400`** (invalid/expired token, password too weak, missing fields):
```json
{ "detail": "Invalid or expired reset link." }
```

> On success, all existing auth tokens for that user are deleted. The user must log in again with the new password.

## What the frontend needs to build

### New pages (Next.js — `frontend-next`)

#### `/auth/forgot-password`
- Single email input field
- On submit: call `authService.requestPasswordReset(email)`
- Always show the same neutral message after submit regardless of what came back: *"If an account with that email exists, a reset link has been sent."*
- No error state needed for "email not found" — see note above

#### `/auth/reset-password`
- Reads `uid` and `token` from URL search params on mount
- If either is missing, show an error and link back to `/auth/login`
- New password input + confirm password input
- On submit: call `authService.confirmPasswordReset(uid, token, newPassword)`
- On success: show confirmation message and redirect to `/auth/login`
- On error: show the `detail` message from the response (e.g. "Invalid or expired reset link.")

---

### New service methods (`frontend-next/src/services/auth.ts`)

Add these two methods to the existing `authService` object, following the same pattern as the other methods in that file:

```typescript
requestPasswordReset: async (email: string) => {
  const response = await authApi.post('/auth/request-password-reset/', { email });
  return response.data;
},

confirmPasswordReset: async (uid: string, token: string, newPassword: string) => {
  const response = await authApi.post('/auth/confirm-password-reset/', {
    uid,
    token,
    new_password: newPassword,
  });
  return response.data;
},
```

Note the field name: `new_password` (snake_case) — that is what the backend expects.

---

### "Forgot password?" entry point

Add a **"Forgot password?"** link on both:
- `frontend-next/src/app/auth/login/page.tsx` — below the password field
- `frontend-next/src/app/auth/signup/page.tsx` — optional, but consistent

Link destination: `/auth/forgot-password`

---

### Middleware (`frontend-next/src/proxy.ts`)

The two new routes are public (no login required). No middleware changes needed — the existing config only blocks the dashboard and profile routes, so `/auth/forgot-password` and `/auth/reset-password` are already open.

However, confirm that `/auth/reset-password` is added to the **auth routes** list (the ones that redirect *away* if the user is already logged in), consistent with `/auth/login` and `/auth/signup`.

---

### Route inventory update (`frontend/src/App.jsx` — React)

If the React frontend also needs these pages, the same pattern applies. Add routes:
- `/auth/forgot-password` → `ForgotPassword` component
- `/auth/reset-password` → `ResetPassword` component

## Full flow end-to-end

```
User on /auth/login clicks "Forgot password?"
  → navigates to /auth/forgot-password

User enters email and submits
  → POST /api/auth/request-password-reset/  { email }
  ← 200  (always)
  → show: "If an account with that email exists, a reset link has been sent."

Backend (invisible to user):
  → looks up account
  → generates uid + token
  → sends email via GAS with link: {FRONTEND_URL}/auth/reset-password?uid=...&token=...

User clicks link in email
  → lands on /auth/reset-password?uid=MQ&token=abc123-xyz
  → page reads uid and token from URL

User enters new password and submits
  → POST /api/auth/confirm-password-reset/  { uid, token, new_password }
  ← 200  → redirect to /auth/login with success message
  ← 400  → show error (link expired, already used, password too weak)
```

---

## Notes

- **Token lifetime:** Django's default is 3 days (`PASSWORD_RESET_TIMEOUT` setting). The GAS email template already tells users *"This link expires in 24 hours"* — if you want to enforce that strictly, set `PASSWORD_RESET_TIMEOUT = 86400` in `backend/lamla/settings.py`.
- **One-time use:** The token is automatically invalidated after a successful reset because the password hash changes. No risk of reuse.
- **Rate limiting:** The request endpoint is throttled at 5 requests/hour per IP (same `AuthThrottle` used across all auth endpoints).
- **No AuthContext changes needed** unless you want to expose the methods there too. The pages can call `authService` directly.
